# 第 1 章：网络基础

本章目标是建立最小但够用的网络模型：一台机器如何找到目标、如何把包发出去、DNS 在哪里参与、出口 IP 为什么会变化，以及为什么做隧道和受控出口前必须先理解路由。

## 1. 这是什么

网络基础不是一组孤立名词，而是一条请求从本机到目标服务的路径。

一个最简化的访问过程通常是：

```text
应用程序
  -> DNS 解析域名
  -> 操作系统选择源 IP、目标 IP、目标端口
  -> 查路由表
  -> 交给网关
  -> 经过运营商、云厂商或中继节点
  -> 到达目标服务
  -> 响应沿路径返回
```

本项目最常用的概念如下。

| 概念 | 作用 |
| --- | --- |
| IP 地址 | 标识网络中的主机或接口 |
| 端口 | 标识主机上的应用服务 |
| DNS | 把域名解析成 IP 地址 |
| 路由 | 决定数据包下一跳发往哪里 |
| 默认网关 | 当没有更具体路由时使用的下一跳 |
| 防火墙 | 控制哪些流量允许进入、离开或转发 |
| 出口 IP | 目标服务看到的来源 IP |
| 隧道 | 把一种网络流量封装进另一条连接里传输 |

## 2. 为什么需要

学习中继节点、加密隧道和受控出口时，最常见的问题不是命令不会敲，而是不知道流量实际走了哪里。

例如：

- 你以为流量走了隧道，但 DNS 仍然从本地网络发出。
- 你以为服务器是出口节点，但默认路由没有切换。
- 你以为防火墙保护了机器，但转发链没有限制。
- 你以为隐藏了本机 IP，但目标服务仍能通过应用层账号、浏览器指纹或行为模式识别你。

网络基础能帮助你把这些问题拆成可检查的事实。

## 3. 它解决什么问题

网络基础能解决的问题：

- 判断当前机器使用哪个出口 IP
- 判断 DNS 请求大概率由谁处理
- 判断访问某个目标时使用哪条路由
- 判断本机有哪些服务正在监听端口
- 判断防火墙是否拦截了入站、出站或转发流量
- 判断隧道、VPN、Exit Node 是否真的影响了默认路由

做任何中继网络实验前，都应该先回答三个问题：

```text
我的流量从哪里出？
我的 DNS 由谁解析？
我的路由表现在是什么状态？
```

## 4. 它不能解决什么问题

网络基础不能直接解决：

- 应用层账号关联
- 浏览器指纹识别
- 目标服务自己的风控策略
- 恶意软件或终端被入侵的问题
- 云厂商、运营商或目标服务的日志留存问题
- 违法或违反服务条款行为带来的风险

加密隧道通常只能保护链路中的部分内容和路径，不等于匿名，也不等于安全。

## 5. 实验步骤

本章对应实验：`labs/lab-01-check-ip-route.md`。

建议在一台 Linux 测试机上执行。不要在生产服务器上随意修改网络配置。

### 5.1 查看本机地址

```bash
ip addr
```

重点观察：

- 哪些网卡是 `UP`
- 哪些网卡有 IPv4 地址
- 哪些网卡有 IPv6 地址
- 是否存在 `wg0`、`tailscale0`、`tun0` 之类的隧道接口

### 5.2 查看默认路由

```bash
ip route
ip -6 route
```

重点观察：

- `default via ... dev ...`
- 默认网关 IP
- 默认出口网卡
- 是否有更具体的网段路由

### 5.3 查看访问某个目标会走哪条路由

```bash
ip route get 1.1.1.1
ip route get 8.8.8.8
```

这类命令不会真正访问目标服务，只是询问内核路由选择结果。

### 5.4 查看公网出口 IP

```bash
curl -4 https://ifconfig.me
curl -6 https://ifconfig.me
```

如果当前机器没有 IPv6，第二条命令可能失败，这是正常现象。

也可以用 DNS 方法查询出口 IP：

```bash
dig +short myip.opendns.com @resolver1.opendns.com
```

### 5.5 查看 DNS 配置

```bash
cat /etc/resolv.conf
resolvectl status
```

不同 Linux 发行版 DNS 管理方式不同。如果没有 `resolvectl`，先记录 `cat /etc/resolv.conf` 的结果即可。

### 5.6 查看路径

```bash
tracepath 1.1.1.1
```

如果安装了 `traceroute`，也可以执行：

```bash
traceroute 1.1.1.1
```

注意：路径探测结果不一定完整。有些网络设备会屏蔽或限速 ICMP、UDP 探测包。

### 5.7 查看监听端口

```bash
ss -tulpen
```

重点观察：

- 哪些端口在监听
- 监听地址是 `127.0.0.1`、`0.0.0.0`、`::` 还是具体内网 IP
- 对应进程名是什么

## 6. 常见坑

### 坑 1：出口 IP 和本机内网 IP 不是一回事

`ip addr` 看到的是本机接口地址，目标服务看到的是公网出口 IP。云服务器通常有内网 IP 和公网 IP，家庭网络通常会经过 NAT。

### 坑 2：DNS 走哪里和网页走哪里可能不同

浏览器、系统、VPN 客户端、Tailscale、容器运行时都可能改变 DNS 行为。检查出口 IP 不等于检查了 DNS 泄露。

### 坑 3：默认路由不是唯一规则

路由表遵循更具体路由优先。例如 `10.0.0.0/8` 的路由会优先于默认路由。

### 坑 4：IPv6 可能绕过你的 IPv4 预期

如果你只配置了 IPv4 隧道或防火墙，但系统仍有可用 IPv6，部分应用可能优先走 IPv6。

### 坑 5：`0.0.0.0` 监听不等于只监听本机

服务监听 `0.0.0.0:PORT` 表示它接受所有 IPv4 接口上的连接。公网服务器上这通常需要特别小心。

## 7. 安全提醒

- 不要把 SOCKS5、HTTP Proxy、WireGuard、SSH 转发端口开放给公网匿名使用。
- 不要在没有回滚方案的 SSH 会话里直接修改默认路由和防火墙。
- 修改防火墙前，先确认云厂商安全组和本机防火墙都允许你的 SSH 管理入口。
- 实验记录不要公开写入真实公网 IP、用户名、密钥、内网拓扑和访问凭据。
- 隧道和 VPN 不是匿名系统。目标服务仍可能通过账号、设备、浏览器、时间模式和行为模式识别访问者。
