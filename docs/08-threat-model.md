# 第 8 章：Threat model

本章建立中继网络实验的威胁模型。目标是明确每个观察者能看到什么、看不到什么，以及你真正想保护的是什么。

## 1. 这是什么

Threat model 是对风险的结构化描述。它不问“是否绝对安全”，而是问：

- 资产是什么？
- 对手是谁？
- 他们能观察什么？
- 他们能控制什么？
- 哪些风险接受，哪些必须降低？
- 失败时如何发现和回滚？

## 2. 为什么需要

如果没有 threat model，很容易把“换出口 IP”误解成“匿名”。

中继网络里至少有这些角色：

- 本地设备
- 本地运营商或 Wi-Fi 网络
- 云厂商
- 中继节点或出口节点
- DNS resolver
- 目标服务
- Tailscale 或其他控制平面
- GitHub 等代码托管平台

每一方看到的信息不同，能力也不同。

## 3. 它解决什么问题

Threat model 能帮助你：

- 明确项目边界
- 选择合适的技术方案
- 解释 DNS、路由、出口和应用层风险
- 避免过度承诺隐私效果
- 设计日志、密钥、ACL 和回滚策略
- 写出更负责任的教程

## 4. 它不能解决什么问题

Threat model 不能替你消除所有风险，也不能证明系统绝对安全。

它不能解决：

- 不可信终端
- 错误操作
- 被入侵的出口节点
- 违法或违规用途
- 目标服务的多维识别
- 第三方平台规则和合规要求

## 5. 实验步骤

### 5.1 定义资产

示例：

```text
资产：
- SSH 私钥
- WireGuard 私钥
- Tailscale tailnet 访问权限
- 内网服务地址
- 实验服务器公网 IP
- DNS 查询记录
- 实验笔记
```

### 5.2 定义观察者

| 观察者 | 可能看到 |
| --- | --- |
| 本地运营商 | 你连接到哪个中继或 VPN 服务器、连接时间、流量大小 |
| 云厂商 | 云服务器元数据、网络流量元数据、磁盘与安全组配置 |
| 中继节点 | 经由它发出的目标 IP、时间、流量大小，未加密应用内容 |
| DNS resolver | 查询域名、查询时间、来源地址 |
| 目标服务 | 出口 IP、账号、Cookie、User-Agent、TLS/HTTP 元数据、行为模式 |
| Tailscale 控制平面 | 设备身份、tailnet 元数据、路由发布状态 |
| GitHub | 你公开提交的教程、提交元数据、可能误提交的敏感信息 |

### 5.3 定义不保护内容

示例：

```text
本项目不承诺：
- 匿名访问第三方服务
- 绕过平台风控
- 隐藏应用层账号身份
- 对抗终端恶意软件
- 对抗所有全球被动观察者
```

### 5.4 定义控制措施

| 风险 | 控制措施 |
| --- | --- |
| 私钥泄露 | 独立密钥、权限 `600`、不提交 Git |
| DNS 泄露 | 使用远端解析、检查 resolver、防火墙约束 |
| 出口误用 | ACL、来源限制、日志、非公共开放 |
| SSH 锁死 | 回滚任务、控制台、保留管理入口 |
| 路由回落 | kill switch、出口 IP 检查、监控 |
| 过度暴露内网 | subnet route 最小化、ACL 最小权限 |

### 5.5 写项目安全声明

README 中应该保留清晰边界：

```text
本项目用于学习 secure networking、traffic isolation、controlled egress。
不用于绕过平台风控、不用于公共代理、不用于违法用途。
```

## 6. 常见坑

- 把 VPN 等同于匿名。
- 只关注 IP，不关注 DNS、账号和浏览器状态。
- 只保护传输链路，不保护终端和密钥。
- 以为云服务器不会记录网络元数据。
- 公开教程时误提交真实实验输出。
- 忘记撤销已不用的 peer、公钥、auth key 或 deploy key。

## 7. 安全提醒

- 公开项目只放教程和脱敏示例。
- 私钥、token、真实公网 IP、内网网段不要进入仓库。
- 对每个出口节点写清楚负责人、用途、允许来源和日志策略。
- 定期审计 Tailscale devices、ACL、GitHub deploy keys、云安全组。
- 一旦发现密钥泄露，立即吊销并重新生成，不要只删除 Git 历史中的文件。
